# CHG-20260124-MVP: JARVIS 真 of MVP実装

> Authority: REFERENCE (Level 2, Non-binding)

## Why
現状のJARVISは優れた理念を持つものの、UIから実際にパイプラインを動かし、その結果を確認するまでの「実働する薄い連動」が不足していました。また、リポジトリ内に `venv` や `.env` が混入するリスクがあり、CI/品質ゲートとの整合性を高める必要がありました。

## What
1. **リポジトリ衛生 (P0):** `venv`, `.env`, 各種生成物を Git 追跡から除外し、`.gitignore` を強化。
2. **GitHub Actions (P1):** `jarvis_dispatch.yml` を更新し、`scripts/run_dispatch.py` を経由して `gh-pages` に成果物をデプロイするフローを確立。
3. **Backend (P1):** `scripts/run_dispatch.py` を新規作成。既存の `jarvis_core` 機能を呼び出し、定められた JSON 契約 (`summary.json`, `progress.json`) で成果物を出力。
4. **API (P1):** `cloudflare-worker.js` を更新。`/api/run` で GitHub Actions をトリガーし、`/api/runs` でデプロイ済みの成果物をプロキシ。
5. **UI (P1):** `dashboard/index.html` を更新。ボタン押下で実行を開始し、進捗をポーリング表示。

## Not Doing
- 大規模な `jarvis_core` 内部ロジックの変更。
- 多様な認証認可の実装（Turnstile と PAT による最小限の構成）。
- データベースの導入（`gh-pages` と JSON ファイルによる静的運用を継続）。

## Acceptance
- [ ] `./venv`, `.env` 等が `git ls-files` に表示されない。
- [ ] `scripts/run_dispatch.py` が正しく JSON を出力する。
- [ ] UI からボタンを押すと GitHub Actions が開始される。
- [ ] 完了後、UI 上に成果物へのリンクが表示される。
- [ ] 既存の全品質ゲート (ruff, black, coverage, security_gate) がパスする。
